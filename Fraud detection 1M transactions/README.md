# Fraud Detection — 1M Transactions (Tier A Diagnostic)

A leakage-honest analysis of a **synthetic** 1,000,000-transaction fraud benchmark, built to the
McKinsey communication standard in [`DOCS/STRUCTURE.md`](DOCS/STRUCTURE.md) and styled per
[`DOCS/DESIGN.md`](DOCS/DESIGN.md).

> **The one fact that shapes everything:** fraud in this dataset is generated by a *known linear
> formula* whose inputs are the same columns you'd model on. Predicting fraud is therefore trivial by
> construction and **any headline score is non-transferable to production.** So this project measures
> **methodology quality**, not detection performance — see [`DOCS/fraud_dataset_analysis.md`](DOCS/fraud_dataset_analysis.md).

---

## Key findings

| # | Finding | Evidence |
|---|---|---|
| 1 | **Fraud is concentrated and predictable** — foreign, high-risk-MCC, unknown-device and overnight transactions each multiply the 1.71% base rate. | `fig_segment_uplift.png`, `fig_hourly_seasonality.png` |
| 2 | **The model recovers the generator, it does not detect fraud** — knocking out `velocity` collapses PR-AUC by **0.22**; ROC-AUC (0.994) vs PR-AUC (0.864) shows why ROC misleads at a 1.7% base rate. | `fig_ablation.png`, `fig_pr_curve.png` |
| 3 | **The network graph is the one independent, transferable signal** — ring features add **+0.0016 PR-AUC** the transaction formula cannot supply. | `fig_ring_lift.png`, `fig_ring_descriptives.png` |
| 4 | **Decision artifact** — ranking by model score, a **2% review budget captures ~82% of fraud**. | `fig_capacity_recall.png` |

**Governing thought:** *fraud here is engineered, not discovered — so the deliverable is a
leakage-honest method plus a review-capacity policy, not a headline score.*

👉 **Full deliverable:** open [`reports/final_report.html`](reports/final_report.html) in a browser
(self-contained, light/dark, figures embedded).

## How to run

```bash
pip install -r requirements.txt          # Python 3.14; pandas 3, xgboost 3, scikit-learn 1.9

# execute the pipeline top-to-bottom (each notebook chains to the next)
cd notebooks
jupyter nbconvert --to notebook --execute --inplace 01_ingestion.ipynb
jupyter nbconvert --to notebook --execute --inplace 02_cleaning.ipynb
jupyter nbconvert --to notebook --execute --inplace 03_eda.ipynb
jupyter nbconvert --to notebook --execute --inplace 04_analysis.ipynb   # ~1.5 min (XGBoost ×11)
jupyter nbconvert --to notebook --execute --inplace 05_reporting.ipynb  # writes final_report.html
```

Everything is reproducible with `seed = 42`. Notebook 04 is the only heavy step (~90s).

## Pipeline (five notebooks = five STRUCTURE.md stages)

| Notebook | Stage | Produces |
|---|---|---|
| [`01_ingestion.ipynb`](notebooks/01_ingestion.ipynb) | Ingestion | schema validation, provenance log |
| [`02_cleaning.ipynb`](notebooks/02_cleaning.ipynb) | Cleaning | dtypes, business rules, **leakage register**, `data/processed/*.parquet` |
| [`03_eda.ipynb`](notebooks/03_eda.ipynb) | EDA | segment uplift, pattern signatures, ring descriptives (4 figures) |
| [`04_analysis.ipynb`](notebooks/04_analysis.ipynb) | Stats + ML | FDR-corrected tests, XGBoost, leakage ablation, ring-lift, capacity curve (4 figures) |
| [`05_reporting.ipynb`](notebooks/05_reporting.ipynb) | Reporting | the McKinsey HTML report |

**Method choices:** temporal train/test split (not random), `scale_pos_weight` over SMOTE, XGBoost,
**PR-AUC + Precision@80%Recall** (not headline ROC-AUC). The leakage register bans
`account_profiles` fraud fields and `fraud_pattern` from all models.

## Limitations (surfaced, not buried)

Labels are instantaneous and certain (no 30–90 day confirmation lag); fraud rings are static; there is
no chargeback/review-cost data, so no Rupiah ROI. **Every performance number is non-transferable by
construction.** A production analogue would owe a subgroup fairness audit and drift monitoring, and
must re-run the ablation on non-leaky, delayed-confirmation labels before any performance claim.

*Synthetic data (`numpy.random.default_rng(seed=42)`) — no real PII.*
